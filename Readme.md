# Настройка `Git` и клонирование проекта

## Установка `Git`

1. Переходим по ссылке и скачиваем установщик [https://git-scm.com/install/windows](https://git-scm.com/install/windows)
2. Запускаем установщик и на всех шагах прожимаем дальше, оставляя паарметры по умолчанию
3. Открываем терминал и вставляем команду `set PATH=c`, чтобы обновить пути системы без перезапуска ПК
4. Закрываем все терминалы, включая все IDE (да, они тоже считаются за терминал)
5. Теперь команда `git` должна быть доступна на ПК
6. Конфигурируем `git` с помощью команд:

```Bash
git config --global user.name "<Имя>"
git config --global user.email "<youremail@example.com>"
## Вместо примеров имени и почты вставляете свои (Имя пишите на английском)
```

## Регистрация в `Github`

Далее необходимо зарегистрировать акаунт на [https://github.com/](https://github.com/). Я думаю, что этот шаг не требует особых пояснений

1. Переходим по ссылке на проект для вашей группы, и находим там кнопочку Fork. Либо к адресу репозитория проекта просто добавляем `/fork` и получаем:
   | Группа | `Fork`-ссылка |
   |--------|---------------|

   | П-318 | https://github.com/TOP-P-318/PersonalAccount/fork |
   | П-319 | https://github.com/TOP-P-319/PersonalAccount/fork |

2. Снимаем галочку копировать только `master` ветку - нам нужны все ветки.
3. Получаем копию репозитория на своем акканте, в которую вы можете вносить правки и отправлять push запросы

## Добавление SSH ключа в `GitHub`

Теперь нам надо затянуть актуальные изменения на свой компьютер.
Мы будем использовать SSH-подключение к `GitHub`, так как оно более надежное, чем Https, и оно используется на реальных проектах. Чтобы подключиться по SSH к репозиторию, для начала необходимо создать ssh-ключ локально на вашем ПК, и затем уже добавить его в `GitHub`

1. В терминале пишем команду

```Bash
ssh-keygen -t ed25519
```

2. Жмакаем несколько раз подряд `Enter` - нам годится дефолтное имя файла, и нам не нужен пароль для доступа к нему
3. Теперь у нас появилась пара файлов в папке `C:\Users\<Имя Пользователя>\.ssh` с названием `id_ed25519`
4. Открываем в блокноте тот. у которого расширение `.pub`, и копируем его содержимое
5. Переходим в `GitHub` `Settings` -> `SSH & GPG kesy` -> кликаем `New SSH key` (или сразу переходим по ссылке [https://github.com/settings/ssh/new](https://github.com/settings/ssh/new))
6. Даем нашему ключу либое имя и вставляем содержимое `id_ed25519.pub` в поле для ввода
   Все, мы связали наш ПК и наш `GitHub`, теперь мы можем подключаться по SSH

## Клонирование репозитория

Теперь нам надо склонировать репозиторий на наш ПК

1. Идем на наш фркнутый репозиторий. Жмем на кнопку `<>Code`, выбираем `SSH` и копируем ссылку на репозиторий
2. Переходим в папку, где хотим, чтобы появилась папка с нашим проектом
3. Клонируем репозиторий. Для этого мы импользуем команду `git clone <SSH ссылка на ваш репозиторий>`
   - При первом подключении по SSH скорее всего возникнет предупреждение о том, что данный ключ не зарегистрирован под другими именами, и предложение продолжить с просьбой ввести слово `yes` для продолжения. Вводим `yes`, и жмем `Enter`

Проект склонирован, можем приступать к выполнению задания

# Обновление состояние своего `Fork`-репозитория от основного репозитория, в котором ведется разработка во время занятий

Дело в том, что после того, как вы сделали свой `Fork`, ваш `Fork`, по сути, стал независимой копией исходного проекта. Он не получает никаких автоматических обновлений кода. Это логично, ведь в `Fork`-репозиторий вы вносите изменения независимо. Часть из этих изменений может конфликтовать с изменениями исходного репозитория, с которого была взята копия. Однако есть вполне простой способ, как актуализировать состояние своего репозитория.

## Настройка `upstream`

На занятии мы с вами настраивали `origin` через `git remote add oring <Url репозитория>`. Тогда я сказал, что `origin` - это просто обычная переменная, в которую мы положили `<Url репозитория>`, и таких переменных может быть сколько угодно, и названия у них могут быть любые (`origin` - это просто название, принятое по умолчанию для удаленного репозитория). Давайте добавим еще одну переменную к торую мы положим ссылку на исходный репозиторий, с которого делался `Fork`. Для удобства назовем ее `upstream`. Ссылки на репозитории для групп будут сделющими:
| Группа | `SSH`-ссылка |
|--------|---------------|
| ПД-412 | git@github.com:TOP-PD-412/PersonalAccount.git |
| П-318 | git@github.com:TOP-P-318/PersonalAccount.git |
| П-319 | git@github.com:TOP-P-319/PersonalAccount.git |

1. Настраиваем `upstream` через команду

```Bash
git remote add upstream "<Url основного репозитрия, на основе которого вы хотите обновить свой Fork-репозиторий>"
```

2. Теперь у нас помимо `origin` есть еще и `upstream`, теперь главное в них не запутаться.
   `origin` указывает на репозиторий, в котором работаете вы, и в котором вы делаете домашние задания,
   `upstream` указывает на репозиторий, в котором мы работаем во время занятий, и из которого вы хотите подтягивать актуальное состояние проекта

Этот шаг нужно было выполнить всего 1 раз на проект. Следующие шаги выполняются в каждом домашнем задании

## Обновление данных из исходного репозитория

1. Сначала нам надо актуализировать информацию о состоянии `upstream`
   для этого выполняем команду

```Bash
git fetch upstream ## теперь надо явно указывать, что именно мы хотим запросить
```

2. В каждом задании, начиная с 4 в самом задании приложена ссылка на ветку, на основе которой необходимо делать задание. На надо затянуть эту ветку к себе в проект

```Bash
git checkout -b lessonX-XX upstream/lessonX-XX ## эту ветку называйте не homework-xxx, а так же, как она названа в исходном репозитории
## очевидно, что lessonX-XX вместо X-XX будет иметь какие-то кокретные числа
```

[ВАЖНО]: Вы начинаете делать задание не с ветки `master` удаленного репозитория, а с той ветки, которая указана в домашней работе. Потому что проект у нас постоянно дорабатывается, и какие-то части домашней работы могут стать неактуальными спустя время

3. Поздравляю, вы только что получили копию ветки с иходного репозитория локально у себя. Теперь надо залить эту ветку к себе в ваш `Fork`-репозиторий

```Bash
git push origin lessonX-XX ## -u можно не писать, вы не будете вносить изменения в эту ветку
```

## Нюансы касательно выполнения домашнего задания

1. Теперь, когда вы приступаете к домашнему заданию, вы создаете новую ветку не от `master`, а от той, которую вы только что затянули

```Bash
git checkout lessonX-XX ## без флага -b. Мы просто удостовериваемся, что сейчас мы находимся на нужной нам ветке
git checkout -b homework-xxx ## а вот уже с этой ветки мы создаем новую, в которй будем делать нашу домашку
```

2. Теперь, когда мы закончили с нашей домашкой, нам необходимо создать `Pull Request`
   Но `PR` мы будем делать уже не в `master`-ветку, а именно в ту, от которой мы создавали ветку `homework-xxx`
   [Важно]: вы создаете `RP` из ветки `homework-xxx` своего репозитория в ветку `lessonX-XX` своего же репозитория.

## Начало работы

1. Создаем новую ветку с номером домашнего задания. Используем команду

```Bash
git checkout -b homework-xxx ## вместо `xxx` - номер ДЗ
```

2. Поскольку мы клонировали репозиторий, то `remote origin` настраивать не нужно, он уже и так прописан. Поэтому просто делаем

```Bash
git push -u origin homework-xxx ## xxx тут тоже, естественно, заменяем
```

Да, мы пушим ветку еще без изменений, чтобы она появилась в нашем репозитории. 3. Переходим на `GitHub` в раздел `Pull Requests` (просто к имени репозитория добавляем `/pulls`)
| Группа | `Fork`-ссылка |
|--------|---------------|
| ПД-412 | https://github.com/TOP-PD-412/PersonalAccount/pulls |
| П-318 | https://github.com/TOP-P-318/PersonalAccount/pulls |
| П-319 | https://github.com/TOP-P-319/PersonalAccount/pulls | 4. Жмем `New pull request`, и меняем `compare` ветку с `master` на ветку с нашей домашкой. Ветку `base` не трогаем - она должна остаться `master` 5. Даем имя нашему PR в виде "Homework XXX", и жмакаем на `Create pull request` 6. Все, теперь все изменения, которые мы будем пушить в эту ветку, будут подтягиваться в PR. 7. По ходу выполнения задания делаем коммиты на каждое ключевое действие

- Создали сервис - `commit`
- Добавили миграцию - `commit`
- Подключили сервис в контроллер - `commit`
- Пофиксили баг - `commit`

Для совершения коммита необходимо выполнить 2 команды:

```Bash
git add . ## добавляет изменения файлов в `staging` (точка важна после add - она означает, добавить все файлы и папки. перед точкой обязательно должен быть пробел)
git commit -m "имя коммита" ## выполняет фиксацию измений, которые добавлены в `staging`
```

8. После этого коммит добавлен локально, теперь его надо отправить в удаленный репозиторий.
   Просто делаем `git push` без каких-то дополнительных действий
9. Проверяем, что на `GitHub` наши изменения действительно попали
